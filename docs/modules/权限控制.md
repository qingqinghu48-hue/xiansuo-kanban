# 权限控制

## 概述
三级角色权限体系，控制数据可见范围和操作权限。

## 核心文件
- `server/middleware/auth.js` — 认证中间件
- `server/routes/auth.js` — 登录/登出/当前用户
- `server/routes/users.js` — 用户管理
- `server/db.js` — 密码哈希（scrypt + salt）

## 角色定义

### admin（管理员）
- 看全部线索和数据
- 可删除/批量删除线索
- 可看成本图表和 KPI 累计数据
- 可管理所有用户（创建/启用/禁用/删除）
- 可修改所有线索的"所属招商"
- 管理后台 4 个 Tab：线索导入、招商线索管理、平台来源管理、用户管理

### agent（招商员）
- 只看自己负责的线索（`所属招商 = 当前用户名`）
- **不可删除**任何线索
- **不可看**成本图表和累计 KPI（前端隐藏）
- 可进入"个人设置"修改自己的姓名和密码

### guest（游客）
- 只读权限，可看全部线索但不可编辑/删除
- 无任何管理权限

## 密码安全
- 使用 Node.js 内置 `crypto.scryptSync` + 随机 salt
- 存储格式：`hex(salt):hex(hash)`
- 启动时自动迁移存量明文密码（`migratePasswords`）
- 登录/改密/创建用户均通过 `verifyPassword`/`hashPassword` 处理

## 会话管理
- Express Session + Cookie
- **Session Store**: 基于 `better-sqlite3` 的 `SqliteSessionStore`，session 持久化到 SQLite `sessions` 表
- **有效期**: 14 天（`rolling: true`，每次请求自动刷新）
- **GC**: 每小时清理过期 session
- **持久化优势**: 服务器重启/重新构建后登录态不丢失
- 前端 `fetch` 必须加 `credentials: 'include'`
- 后端 CORS 配置 `credentials: true`

## 实时用户校验
`requireAuth` / `requireAdmin` / `current_user` / `change-password` 每次请求都查数据库验证：
- 用户被删除 → 销毁 session，返回 401
- 用户被停用 → 销毁 session，返回 401
- 数据库异常时非严格模式允许通过（避免系统完全不可用）
